# סביבת הבדיקות (Testing & E2E)

מסמך זה מסביר איך עובדת סביבת הבדיקות של הפרויקט, כדי שכל מפתח (אנושי או AI) יוכל להבין ולתחזק אותה.

## העיקרון

לפני מסירת כל פיצ'ר מבצעים בדיקת **End-to-End** אמיתית עם **משתמש בדיקה ייעודי** — התחברות, פעולה, שמירה ל-DB, רענון מלא, וקריאה מחדש מה-Database (לא רק מה-state המקומי).

## משתמש הבדיקה

- אימייל ייעודי ומסומן: `e2e-test@firm.local`.
- **מבודד לחלוטין** מהנתונים האמיתיים: Supabase RLS מסנן כל שורה לפי `user_id`, אז משתמש הבדיקה מתחיל ריק ולא רואה/נוגע בלקוחות האמיתיים.
- ההתחברות בפיתוח נעשית דרך כפתור **"🧪 כניסה כמשתמש בדיקה"** במסך ההתחברות (מוצג רק ב-DEV), שמשתמש בפרטים מ-`.env.local`.

## הסקריפט המרכזי

`scripts/setup-test-env.mjs` — מקור האמת היחיד לסביבת הבדיקות.

```bash
npm run test:setup     # ודא/שחזר את משתמש הבדיקה + כתוב פרטי כניסה ל-.env.local
npm run test:status    # בדוק אם משתמש הבדיקה קיים
npm run test:reset      # מחק את נתוני משתמש הבדיקה בלבד (דורש --yes)
```

תכונות:
- **idempotent** — אפשר להריץ שוב ושוב. בודק קודם אם המשתמש קיים, יוצר רק אם לא; אם קיים — מסנכרן סיסמה ומוודא שהוא מאושר.
- **סיסמה אקראית חזקה** נוצרת אוטומטית ונכתבת ל-`.env.local` — שום סיסמה לא עוברת ידנית.
- **שחזור עצמי** — אם משתמש הבדיקה נמחק/נשבר, הרצה חוזרת מקימה אותו מחדש.

## אבטחה

- ה-**service_role key** נשמר **אך ורק** ב-`.env.local` (שמוסתר מ-git). לעולם לא נכנס ל-Git/GitHub/לוגים/קוד הלקוח.
- שם המשתנה **בלי** הקידומת `VITE_` — ולכן Vite **לעולם לא מכניס אותו** ל-bundle של הדפדפן. רק סקריפט הצד-שרת קורא אותו.
- האפליקציה רואה רק את `VITE_DEV_USER_EMAIL/PASSWORD` (פרטי משתמש הבדיקה), לעולם לא את ה-service_role.

## הגנות (DEV בלבד)

- הסקריפט מסרב לרוץ עם `NODE_ENV=production`.
- פועל אך ורק על משתמש הבדיקה הייעודי; פעולות מחיקה מוגבלות לשורות עם ה-`user_id` שלו בלבד, ודורשות `--yes`.

> ⚠️ **הערה ארכיטקטונית:** כרגע יש פרויקט Supabase **אחד** המשמש גם לפיתוח וגם לייצור. הבידוד מתבסס על משתמש הבדיקה (RLS), לא על DB נפרד. לטווח ארוך, סביבת DEV נפרדת (פרויקט Supabase נפרד או Supabase Branch) תיתן הפרדה אמיתית — ראה "צעדים הבאים".

## משתני סביבה נדרשים (`.env.local`)

ראה `.env.local.example`. בקצרה:
```
VITE_SUPABASE_URL=...
VITE_SUPABASE_ANON_KEY=...
SUPABASE_SERVICE_ROLE_KEY=...        # שרת בלבד — בלי VITE_, לא נכנס ל-bundle
VITE_DEV_AUTO_LOGIN=true             # נכתב ע"י הסקריפט
VITE_DEV_USER_EMAIL=e2e-test@firm.local   # נכתב ע"י הסקריפט
VITE_DEV_USER_PASSWORD=...           # נכתב ע"י הסקריפט (אקראי)
```

## תהליך ה-E2E לפני כל מסירה

1. התחברות כמשתמש הבדיקה.
2. ניווט למסך הרלוונטי.
3. ביצוע הפעולה כמו משתמש אמיתי.
4. שמירה ל-DB.
5. רענון מלא של הדף.
6. אימות שהמידע נקרא מחדש מה-DB (לא רק מ-state).
7. ללא שגיאות Console.
8. ללא שגיאות Network.
9. אימות מבנה הנתונים במסד (כשאפשר).
10. אימות שאין Regression לפונקציונליות קיימת.

## צעדים הבאים (שכבות שנוסיף)

- **Playwright + Storage State** — חבילת בדיקות regression אוטומטית: התחברות חד-פעמית כמשתמש הבדיקה, שמירת ה-session (storageState), והרצת תרחישים חוזרים בלי התחברות מחדש.
- **Fixtures / Seed** — נתוני זרע עקביים למשתמש הבדיקה לתרחישים מורכבים (`--seed`).
- **סביבת DEV נפרדת** — פרויקט Supabase נפרד או Branch לבידוד אמיתי בין פיתוח לייצור.
