# פרומט לקלוד קוד — שדרוג מערך ניהול הלקוחות

> **איך להשתמש בקובץ הזה:** העתק את הכל מהכותרת "תקציר הבקשה" ולמטה ושלח לקלוד קוד. אם אתה רוצה לפצל לעבודה בשלבים, תעתיק רק את "שלב 1" (או "שלב 2" וכו'), כל שלב נבנה כך שיעבוד גם לבד.

## תקציר הבקשה

אני בונה CRM אישי למשרד רו"ח שלי (פרויקט ב-React + Vite + TypeScript, RTL בעברית, אחסון מקומי בלבד). מערך **המשימות עובד טוב** — עכשיו אני רוצה לשדרג את **מערך הלקוחות** מהבסיס. לפני שאתה כותב שורת קוד אחת — קרא את הקבצים הבאים בסדר הזה:

1. `CLAUDE.md` — הוראות העבודה איתי. שים לב במיוחד לסעיף 1 (חובת QA לפני דיווח "סיימתי") ולסעיף 0 (תמיד עברית, בלי ז'רגון טכני).
2. `SPEC.md` — האפיון המלא של הפרויקט.
3. `src/types/index.ts` — מבנה הנתונים הקיים. אני בונה על מה שיש, לא מקפיץ הכל מאפס.
4. `src/components/ClientForm.tsx` ו-`src/components/ClientList.tsx` — המסכים שצריך לשדרג.
5. `src/components/DocumentManager.tsx` — קומפוננט מסמכים קיים, אנחנו ניבנה עליו.

**אל תכתוב קוד לפני שקראת את הקבצים האלה ולפני שיש לך תמונה ברורה של המצב הקיים.** אם משהו מהפרומט הזה סותר את מה שאתה רואה בקוד או נראה לך שגוי — עצור ותשאל אותי, אל תנחש.

## למה אני עושה את השינוי

היום מסך הלקוח שלי הוא בעיקר "טופס ארוך לחישוב מס" — 9 לשוניות (אישי / סיווג מס / משפחה / בן זוג / ילדים / זיכויים / נכסים / הערות / משימות). זה עבד כשהאפליקציה הייתה רק מחשבון מס. עכשיו זה CRM — אני צריך תיק שבו אני רואה במבט אחד **איך הלקוח הזה משלם מס בפועל**: באיזה תדירות הוא מדווח מע"מ, מתי המקדמה הבאה, באיזה פקיד שומה הוא מטופל, איזה מסמכים הוא העלה, אילו משימות פתוחות עליו.

**ההשראה שלי** היא [לקוחות פלוס (cplus.co.il)](https://www.cplus.co.il/) — מבחינת הפיצ'רים, לא מבחינת ה-UX. הם בנויים סביב הקונספט הנכון לארץ: לכל לקוח יש 3 "תיקים" נפרדים מול הרשויות (מ"ה / מע"מ / ניכויים), כל תיק עם מספר משלו, פקיד שומה אחר ותדירות דיווח עצמאית. גם שדות פנימיים כמו "הרשאת חיוב בשע"מ" (טופס 4480) הם חלק מהמודל.

## עקרונות מנחים לכל העבודה

1. **מודל הנתונים מנצח את ה-UI.** אם המודל נכון, ה-UI מתעדכן בקלות. אם המודל שטוח, נשארנו עם form ענקי.
2. **לא למחוק שדות שכבר עובדים.** ה-`Client` interface כיום עשיר. השינויים הם **תוספת + ארגון מחדש של ה-UI**, לא מחיקה. תוסיף מיגרציה לכל לקוח קיים שתעניק לו ערכי ברירת מחדל לשדות החדשים.
3. **3 לשוניות ב-ClientForm במקום 9.** איחוד מסיבי. אבל בתוך הלשוניות יהיו sections ברורים.
4. **כל מה שאתה משנה — תבדוק בדפדפן עם screenshots.** קרא את CLAUDE.md סעיף 1 לפני שאתה מתחיל.

## שלב 1 — מודל נתונים: הוספת ישויות חדשות ושדות תפעוליים

צור entity חדש `TaxFile` ו-entity חדש `Document`. **אל תיגע ב-`Client` interface חוץ מהוספות מינימליות — הקישור ל-tax files יהיה דרך טבלה נפרדת, לא embedded ב-Client.**

### 1.1 ישות `TaxFile` — תיק מול רשות

```ts
export type AuthorityFileKind = 'incomeTax' | 'vat' | 'withholding';

export type ReportingFrequency =
  | 'monthly'        // חודשי
  | 'bimonthly'      // דו-חודשי
  | 'semiannual'     // חצי-שנתי
  | 'annual'         // שנתי
  | 'none';          // לא מדווח

export type VATReportType = 'regular' | 'detailed' | 'none'; // רגיל / מפורט (PCN874)

export type IncomeTaxAdvanceFrequency = 'monthly' | 'bimonthly' | 'none';

/** סוג ניכוי — מה הלקוח מנכה במקור */
export type WithholdingType =
  | 'employees'        // עובדים
  | 'suppliers'        // ספקים
  | 'rent'             // שכ"ד
  | 'professionals'    // נותני שירותים (יועצים, רו"ח, עו"ד)
  | 'other';

export interface TaxFile {
  id: string;
  clientId: string;                    // הלקוח המייצג (תמיד יש)
  kind: AuthorityFileKind;             // איזה תיק (מ"ה/מע"מ/ניכויים)
  isActive: boolean;                   // האם התיק פעיל (אפשר לסגור תיק ולשמור היסטוריה)

  fileNumber: string;                  // מספר התיק/עוסק
  assessmentOfficeId: string;          // id של פקיד השומה (ראה רשימה למטה)

  reportingFrequency: ReportingFrequency;

  // ── שדות ספציפיים לפי סוג ──
  // למ"ה:
  advanceFrequency?: IncomeTaxAdvanceFrequency;
  advanceRate?: number;                // שיעור המקדמה ב-%
  // למע"מ:
  vatReportType?: VATReportType;       // רגיל / מפורט

  // לניכויים:
  withholdingTypes?: WithholdingType[];
  niWithholdingRequired?: boolean;     // חייב בניכויי ב"ל
  niAdvanceAmount?: number;            // סכום מקדמת ב"ל (חודשית/דו-חודשית)

  // ── הרשאות שע"מ ──
  shaamAuthorized: boolean;            // יש הרשאת חיוב בשע"מ (טופס 4480)
  shaamAuthorizedDate?: string;        // מתי ניתנה
  shaamExpiryDate?: string;            // תוקף (אם רלוונטי)

  notes: string;                       // הערות ספציפיות לתיק הזה
  createdAt: string;
  updatedAt: string;
}
```

**נקודות חשובות:**
- `TaxFile` נשמר ב-localStorage תחת `crm_tax_files` (עם `useLocalStorage`).
- שאיבת התיקים של לקוח: `taxFiles.filter(f => f.clientId === client.id)`.
- לקוח יכול להיות בלי תיקים (לקוח חדש שלא נפתחו לו עדיין), או עם 1-3 תיקים. **לא חובה שיהיו לו את שלושת התיקים.**
- בן/בת הזוג של הלקוח **לא מקבלים TaxFiles** — אלא אם הם פתחו להם תיק לקוח עצמאי במערכת. במקרה כזה זה לקוח (`Client`) נפרד עם `id` משלו.

### 1.2 ישות `Document` ו-`DocumentCategory`

```ts
export type DocumentCategory =
  | 'income_tax'         // מס הכנסה
  | 'vat'                // מע"מ
  | 'national_insurance' // ביטוח לאומי
  | 'power_of_attorney'  // ייפויי כוח
  | 'identity'           // תעודות מזהות
  | 'financial'          // דוחות כספיים, מאזנים
  | 'other';             // אחר

export interface Document {
  id: string;
  clientId: string;
  category: DocumentCategory;
  taxFileId?: string;          // אופציונלי — קישור לתיק ספציפי
  taxYear?: number;            // אופציונלי — שנת מס רלוונטית
  fileName: string;
  fileSize: number;
  mimeType: string;
  storedDocId: string;         // id ב-IndexedDB (כמו ב-DocumentManager הקיים)
  tags: string[];              // תגיות חופשיות
  notes: string;
  uploadedAt: string;
}
```

**נקודות חשובות:**
- שמור ב-localStorage תחת `crm_documents` (metadata בלבד); תוכן הקבצים נשאר ב-IndexedDB באמצעות ה-hook הקיים `useIndexedDB`.
- חזור על הסכמה הקיימת ב-`DocumentManager.tsx` ולא תמציא הכל מאפס — תרחיב/תתאים. אם השכבה הקיימת לא מספיקה, **תאמר לי מה צריך לשנות לפני שמשנים**.

### 1.3 רשימת פקידי שומה

צור קובץ חדש `src/data/assessmentOffices.ts` עם רשימה מובנית של פקידי השומה בארץ. דוגמת מבנה:

```ts
export interface AssessmentOffice {
  id: string;
  name: string;            // למשל "פקיד שומה ת"א 1"
  city: string;
  type: 'incomeTax' | 'vat' | 'withholding' | 'all';
}
```

**משימה:** חפש את הרשימה העדכנית באתר רשות המסים או ממקור אמין אחר. אם אינך מצליח לאתר רשימה מלאה — תיצור רשימה חלקית (לפחות 30 פקידי שומה הנפוצים: ת"א 1-5, ירושלים 1-3, חיפה 1-2, באר שבע, נתניה, פתח תקווה, רחובות, רמלה, חולון, אשקלון, כפר סבא, אזור, גוש דן, כרמיאל, נצרת, חדרה, עפולה, צפת, אילת, רעננה, הרצליה, רמת גן) ו**תאמר לי שזו רשימה חלקית** כדי שאדע להשלים. למע"מ ולניכויים יש פקידי שומה משלהם — תכלול אותם או תהיה ערוך לתת אופציה לבחור פקיד שומה זהה לכל סוגי התיקים.

## שלב 2 — עיצוב מחדש של `ClientForm`

המבנה החדש: **3 לשוניות ראשיות + הערות "lateral" צד.**

> רכיב הערות (`notes`) — לא לשונית נפרדת. תהיה תיבה תמידית בצד או בכותרת תמיד נגישה (לבחור: side-bar בתוך הטופס, או footer קבוע, או popover קטן בכותרת). **בחר מה שנראה לך הכי טבעי, ובוא תראה לי screenshot של שתי גרסאות ואני אבחר.** הערות חשובות מאוד אבל לא צריכות לשונית שלמות.

### לשונית 1: "פרטים אישיים" — `tab=personal`

מבנה: שני בלוקים זה לצד זה (במידה ואין בן/בת זוג, רק הבלוק השמאלי מוצג רוחב מלא).

**בלוק שמאל — "הנישום המייצג":**
- שם פרטי, שם משפחה (חובה)
- ת.ז.
- תאריך לידה
- מין
- טלפון, מייל
- עיר מגורים, כתובת מלאה
- מצב משפחתי

**בלוק ימין — "בן/בת הזוג"** (מוצג רק אם `familyStatus === 'married'`):
- שם פרטי, שם משפחה
- ת.ז.
- תאריך לידה
- מין (ברירת מחדל הפוך מהנישום)
- טלפון
- שדה checkbox: "האם יש לבן/בת הזוג תיק נפרד במערכת?" — אם מסומן, הצג כפתור "פתח/קישר תיק לקוח עצמאי" שיוצר Client חדש מקושר.

**מתחת לשני הבלוקים** — תיבה "הערות אישיות" (אם בחרת בארכיטקטורה של notes-בצד, פסול את זה).

### לשונית 2: "נתוני מס וב"ל" — `tab=tax`

זו הלשונית **הכי חשובה** ובה כל מה שקשור למיסוי הלקוח. סדר ה-sections מלמעלה למטה:

#### Section A: סיווג מס
- סיווג מ"ה: שכיר / עצמאי / שכיר+עצמאי / שכירות בלבד / אחר (קיים — `incomeTaxType`)
- סיווג מע"מ: עוסק מורשה / עוסק פטור / לא רשום (קיים — `vatStatus`)
- סיווג ב"ל: עובד שכיר / עצמאי עונה להגדרה / עוסק שאינו עונה להגדרה / שכיר+עצמאי / פסיבי / פנסיונר (קיים — `niType`)
- תיאום מס (קיים — `hasTaxCoordination` + `taxCoordinationDetails`)
- פטור מניכוי במקור (קיים — `hasExemptFromWithholding`)

#### Section B: תיקים מול הרשויות
תצוגה כקלפים — שלושה כרטיסי תיקים: מ"ה / מע"מ / ניכויים. כל קלף:
- אם התיק לא קיים: כפתור "+ פתח תיק [סוג]"
- אם קיים: מציג מספר תיק, פקיד שומה, תדירות דיווח, סטטוס שע"מ — וכפתור "ערוך"
- לחיצה על "ערוך" פותחת מודל עם כל שדות ה-`TaxFile` הרלוונטיים לסוג.

תצוגה מומלצת: 3 כרטיסים בשורה (במסכים רחבים), אחד מתחת לשני (במובייל).

#### Section C: ילדים
(קיים — `children`. תשמור את האותו רכיב.)

#### Section D: נקודות זיכוי
- עולה חדש / תושב חוזר (קיים)
- נכות (קיים)
- תואר אקדמי (קיים)
- שירות צבאי / לאומי (קיים)
- ישוב מזכה (קיים)

> שמור על מבנה ה-collapse בתוך הלשונית — Section שמתקפל כשלא בשימוש כדי לא להעמיס.

#### Section E: נכסים ופנסיה
- דירת מגורים, מס' דירות (קיים)
- פנסיה — קופות גמל, קרן השתלמות (קיים)

#### Section F: בן/בת הזוג — נתוני מס
אם נשוי, הצג section נפרד עם נתוני המס של בן/בת הזוג (מבנה זהה ל-Sections A-E אבל קומפקטי יותר). `SpouseData` הקיים כבר מכיל את כל השדות. **לא להציג** Section B (תיקים) לבן הזוג — תיקים נשמרים רק לנישום המייצג, אלא אם פתחת לבן הזוג Client נפרד.

### לשונית 3: "מסמכים" — `tab=documents`

הצג את כל המסמכים של הלקוח, מקובצים ב-7 הקטגוריות:
- מס הכנסה
- מע"מ
- ביטוח לאומי
- ייפויי כוח
- תעודות מזהות
- דוחות כספיים
- אחר

מבנה כל קטגוריה: כותרת + ספירת קבצים + accordion עם רשימת הקבצים. כל קובץ: שם, גודל, תאריך העלאה, תגיות, כפתורי "פתח/הורד/מחק/ערוך מטא".

מעל הקטגוריות:
- כפתור "העלה מסמך" — פותח מודל עם שדות: קטגוריה (חובה), קובץ (חובה), שנת מס (אופציונלי), קישור לתיק (אופציונלי), תגיות (free text), הערות.
- שורת חיפוש שמסננת לפי שם/תגית.
- פילטר לפי שנת מס.

**שמור על תאימות לאחור עם `DocumentManager.tsx` הקיים** — אם יש כבר מסמכים בלי קטגוריה, תיתן להם `category: 'other'` במיגרציה.

### לשונית 4: "משימות" — `tab=tasks`

(אם בחרת ב-3 לשוניות זה נכלל בלשונית מסמכים-משימות משולבת. **המלצתי הברורה: להשאיר 4 לשוניות עם משימות נפרדת**, כי זה איזור פעיל).

הצג את כל המשימות שמקושרות ל-`clientId` של הלקוח הנוכחי (מ-`crm_tasks`). השתמש ברכיב `TaskCard` הקיים. בראש הלשונית כפתור "+ משימה חדשה לללקוח" שפותח את `TaskForm` עם `clientId` ממולא מראש.

ארגון: שלושה sections — חדשות / בתהליך / הושלמו. תספור ותציג badge.

## שלב 3 — שינויים נלווים

### 3.1 רשימת הלקוחות (`ClientList.tsx`)
הוסף עמודות חדשות לטבלה:
- **סטטוס תיקים** — באדג'ים קטנים שמראים אילו תיקים פתוחים: מ"ה / מע"מ / ניכויים (כל אחד באדג' כשמוצג, אפור מומלץ כשלא קיים)
- **משימות פתוחות** — מספר (אם 0, להציג "—")
- **מסמכים** — מספר כולל

תוכל להחליף עמודה קיימת ב-`ClientList` (כמו "פעולות") כדי לא להעמיס.

### 3.2 כפתור "מחשבון מס" בתיק הלקוח
**להשאיר את הכפתור בתיק הלקוח, אבל לעשות אותו משני** (לא ירוק עיקרי). לדוגמה: כפתור outline אפור/לבן ליד כפתור "שמור". הוסף בנוסף — בלשונית "מדריך מס" (`TaxReferencePanel.tsx`) — כפתור "פתח מחשבון" שאחריו רשימת בחירה של לקוח. כשבוחרים לקוח, נטענים נתוניו אל המחשבון (כמו שעובד היום בכפתור הקיים).

### 3.3 מיגרציית נתונים (חשוב!)
ב-`useLocalStorage` של `crm_clients`, הוסף migration step:
- כל לקוח קיים — אם יש לו `incomeTaxType !== 'employee'`, צור עבורו `TaxFile` של מ"ה (אבל ריק — בלי מספר תיק, בלי פקיד שומה — כדי שגיא ימלא ידנית).
- כל לקוח עם `vatStatus !== 'none'` → יצירת `TaxFile` של מע"מ.
- אל תשמור `TaxFile` של ניכויים אוטומטית — זו פעולה אקטיבית של גיא.

מסמכים קיימים — תן להם `category: 'other'`.

### 3.4 נתוני דוגמה
